# ADR-0001: ストレージ永続化戦略：ローカルファイル保存への限定

## 1. Context (背景・課題)
*   **現状**: Kura-Tas は現在、ブラウザの FileSystem Access API を利用してローカルのファイル（JSON/Markdown）を直接読み書きして動作しています。
*   **課題**: マルチデバイス間での同期やバックアップの観点から、Google Drive 等のクラウドストレージや、ユーザー主権型データストア（Solid、remoteStorage等）への対応可能性を検討しました。
*   **検討した選択肢**:
    1.  **Google Drive などのクラウド API 直接連携**: Google API をブラウザから直接叩いてファイルを操作する。
    2.  **ユーザー主権型ストレージ (Solid, remoteStorage) 連携**: ユーザーが自分でホストする Pod や WebDAV と標準プロトコルで通信する。
    3.  **OSのクラウド同期機能の活用**: アプリ側には手を入れず、OSのデスクトップ同期クライアント（Google Drive パソコン版、Dropbox、OneDrive 等）のフォルダをアプリの保存先に指定してもらう。
    4.  **同期サーバーの導入 (Local-first + CRDT)**: IndexedDB 等をマスターとして、バックグラウンドで自前の同期サーバーと同期する。

## 2. Decision (決定)
*   **決定内容**:
    Kura-Tas 自体にはクラウド API や同期サーバーとの通信ロジックを実装せず、**「ローカルファイル保存（FileSystem Access API）」のみに対応する方針を維持する**。クラウド同期を行いたい場合は、**選択肢3（OS of クラウド同期フォルダの活用）** を推奨する。
*   **決定理由**:
    1.  **セッション寿命とUXの破綻**: サーバーレス（フロントエンドのみ）で Google Drive API 等を直接叩く場合、セッション（アクセストークン）の維持が難しく、1時間ごとに再認可のポップアップ等が発生して利便性が著しく低下する。
    2.  **`file://` スキームでの動作制約**: Kura-Tas の強みである「単一HTMLファイルをローカルでダブルクリックして動かす」というユースケースでは、セキュリティの制限から Google OAuth 等の外部認証が動作しない。
    3.  **セキュリティリスクの回避**: フロントエンドのみのアプリに永続的な認証情報（リフレッシュトークンなど）を保管することは、XSS脆弱性への曝露などセキュリティ上のリスクを伴う。
    4.  **シンプルさの維持（オーバーエンジニアリングの回避）**: ネットワーク切断時の挙動や、複数端末からの同時書き込みによる競合（コンフリクト）を解決する同期エンジンを自作することは、本プロジェクトの規模に対してオーバーエンジニアリングである。

## 3. Consequences (影響・帰結)
この決定によってもたらされる影響（トレードオフ）です。

*   **Pros (メリット)**:
    *   **極めてシンプルなコードベース**: 認証ロジック、通信エラー処理、競合解消ロジックが不要になり、[storage.ts](../../src/core/storage.ts) はローカル IO に専念できる。
    *   **完璧なプライバシーとセキュリティ**: アプリがユーザーのアカウント情報やトークンを一切触らないため、安全性が高い。
    *   **ポータビリティの維持**: `dist/index.html` 1ファイルだけでローカルで動作するポータビリティが保たれる。
*   **Cons (デメリット・制約)**:
    *   **スマートフォン/タブレットでの同期制限**: OS側のフォルダ同期クライアントが使えないモバイルデバイスのブラウザ環境では、PC版とのデータ同期が困難になる。
