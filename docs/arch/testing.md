# テスト戦略 (Testing)

## 1. 配置ルール
テストコードは、対象ソースコードと同じディレクトリ内の `__tests__` フォルダに配置する（Colocation）。
- **例**: `src/core/storage.ts` -> `src/core/__tests__/storage.test.ts`
- **命名**: `[対象ファイル名].test.ts`

## 2. テストの分類
1. **Unit Test**: `engine`, `shared`, `utils` 等の純粋関数。副作用なし。
2. **Integration Test**: `storage`, `store` 等の永続化・状態管理。モックを用いた非同期検証。
3. **UI Test**: `ui.ts` (HTML生成), `renderer.ts` (DOM反映)。`jsdom` を使用。

## 3. 依存関係の遵守
テスト実装においても[依存関係ルール](dependency.md)で定義されたルールを厳守する。

## 4. 堅牢なテストのための原則
以下のテスト設計指針を遵守する。

### 1. モックの境界とカプセル化 (Interface-based Mocking)
- **原則**: テスト対象に DI（依存性注入）で渡されるインターフェース（各 Store など）をモックの境界とする。
- **理由**: Store が内部で利用している下位モジュール（`@/core/handle` や `@/core/storage` など）をテスト側で直接モックすることは避ける。
- **手法**: テストコード内でモック用インスタンスを生成し、対象のメソッドを `vi.spyOn` することで、Store の内部実装が変更（データの取得先が IndexedDB から FileSystem に変わるなど）されてもテストが壊れないようにし、リファクタリング耐性を高める。

### 2. 外部環境（時刻・ランダム値）の制御 (No Flaky Tests)
- **原則**: 実行タイミングによって結果が変わる要素（システム時刻、乱数など）に依存するロジックをテストする場合は、必ずそれらをモック化して値を固定する。
- **手法**: `datetime.getTodayStr()` などの戻り値を `vi.mock` や `vi.useFakeTimers()` で固定値にする。これにより、深夜 0 時付近でのテスト実行失敗（Flaky Test）を防止する。

### 3. 重要な副次作用の明示的検証 (Verify Side Effects)
- **原則**: 単に状態（State）が更新されたことだけでなく、`save` 処理などの重要な副作用が、正しいタイミングと正しい引数で呼び出されたことを検証する。
- **手法**: `expect(store.handle.save).toHaveBeenCalledWith(expected)` のように、重要な永続化処理の呼び出しをアサーションに含める。

## 5. カバレッジポリシー (Coverage Policy)
主要なビジネスロジックの信頼性を担保するため、以下のカバレッジ運用を行う。

### 1. 測定対象の絞り込み
- **対象 (Target)**: `logic.ts`, `Store`, `Engine`, `Utils`。これらはシステムの核心であり、ステートメントカバレッジ 90% 以上を維持する。
- **除外 (Exclude)**: `renderer.ts`, `ui.ts` (HTML生成), `main.ts`, `types.ts`。
- **理由**: View に近い層を含めると、ビジネスロジックの網羅性が表示のテスト不足によって埋もれてしまい、指標としての健全性が損なわれるため。まずは「ロジックの正しさ」を数値で保証することを最優先とする。

### 2. 異常系・エッジケースの優先
- カバレッジの数値向上だけでなく、I/O エラー、トランザクションの失敗、Undo/Redo 時のロールバックなど、異常系のパスをテストすることを重視する。
